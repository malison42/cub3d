Ваша программа должна принимать первым аргументом файл описания сцены с расширением
.cub.

◦ Карта должна состоять только из 6 допустимых символов: `0` — пустая клетка,
`1` — стена, и `N`, `S`, `E`, `W` — позиция спавна игрока и его ориентация.

Пример простой корректной карты:

```
111111
100101
101001
1100N1
111111
```

◦ Карта должна быть замкнута/окружена стенами; если это не так, программа должна
вернуть ошибку.

◦ За исключением содержимого самой карты, каждый тип элемента в файле может
быть отделён одной или несколькими пустыми строками.

◦ За исключением карты (которая всегда должна идти последней), другие типы
элементов могут располагаться в файле в любом порядке.

◦ За исключением карты, части информации элемента могут разделяться одним или
несколькими пробелами.

◦ Карта должна парситься в том виде, в котором она записана в файле. Пробелы
являются допустимой частью карты — вам нужно корректно их обрабатывать. Вы
должны уметь парсить любую карту, пока она соответствует правилам.

### Политика обработки пробелов (строгое правило)

В этом проекте принята жёсткая политика: любой пробел (` `) в карте, который
находится в соседстве с проходимой клеткой (`0`, `N`, `S`, `E`, `W`), считается
признаком незамкнутости карты — то есть ошибка.

Рекомендуемая реализация проверки под эту политику:

1. При чтении блока карты найдите максимальную длину строки и дополните более
	 короткие строки пробелами справа, чтобы получить прямоугольную сетку символов.
2. Трактовка символов:
	 - `1` — стена (непроходимая);
	 - ` ` (пробел) — пустое/незаполненное место;
	 - `0`, `N`, `S`, `E`, `W` — проходимые внутренние клетки.
3. Проверка корректности карты (алгоритм):
	 - После паддинга пройдите по всем клеткам карты.
	 - Если для любой проходимой клетки (`0` или `N/S/E/W`) найдётся сосед (по 4
		 направлениям) с символом ` ` (пробел), карта считается незамкнутой и неверной.

Псевдокод:

```
pad_lines_to_width(map);
for (y = 0; y < height; ++y)
	for (x = 0; x < width; ++x) {
		if (map[y][x] in { '0','N','S','E','W' }) {
			for (each neighbor nx,ny of (x,y))
				if (map[ny][nx] == ' ') return INVALID_MAP;
		}
	}
return VALID_MAP;
```

Пояснение: эта политика проще и строже, она автоматически отвергает любые
встречающиеся вблизи проходимых клеток пробелы — даже если пробелы внутренние и
формально могли бы оказаться не связанными с внешней областью. Такая логика
устраняет неоднозначности и делает поведение парсера более предсказуемым.

Если потребуется более мягкая политика (только внешние пробелы от границы
считаются утечкой), можно вернуться к варианту с паддингом и BFS от границ —
но это будет менее строгая проверка.

◦ За исключением карты, каждый элемент должен начинаться с его идентификатора
(состоящего из одного или двух символов), после которого следует его специфичная
информация в строгом порядке:

* Текстура северной стены:

```
NO ./path_to_the_north_texture
```

	- идентификатор: `NO`
	- путь к файлу текстуры северной стороны

* Текстура южной стены:

```

```

	- идентификатор: `SO`
	- путь к файлу текстуры южной стороны

* Текстура западной стены:

```

```

	- идентификатор: `WE`
	- путь к файлу текстуры западной стороны

* Текстура восточной стены:

```

```

	- идентификатор: `EA`
	- путь к файлу текстуры восточной стороны

* Цвет пола:

```

```

	- идентификатор: `F`
	- цвет в формате `R,G,B`, где значения в диапазоне [0,255]

* Цвет потолка:

	- идентификатор: `C`
	- цвет в формате `R,G,B`, где значения в диапазоне [0,255]

## Пункты, чётко прописанные в правилах

- Файл сцены должен передаваться первым аргументом и иметь расширение `.cub`.
- Карта может содержать только символы `0`, `1`, `N`, `S`, `E`, `W`.
- `0` — пустая клетка, `1` — стена, `N/S/E/W` — стартовая позиция и ориентация
	игрока.
- Карта должна быть замкнута (окружена стенами).
- Карта должна идти последней в файле.
- За исключением карты, элементы могут отделяться одной или несколькими пустыми
	строками.
- За исключением карты, части информации элемента могут разделяться одним или
	несколькими пробелами.
- Все элементы (кроме карты) должны начинаться с идентификатора: `NO`, `SO`,
	`WE`, `EA`, `F`, `C`, после которого следует конкретная информация в строгом
	порядке (путь к текстуре или цвет в формате `R,G,B`).
- Цвета задаются в формате R,G,B, где каждое значение — целое число в
	диапазоне [0,255].
- При любой ошибке парсинга/конфигурации программа должна вывести
	`Error\n` и сообщение об ошибке.

## Пункты, которые могут подразумеваться (рекомендуемые/логические допущения)

- Должна быть ровно одна стартовая позиция игрока (`N`, `S`, `E` или `W`).
	(Правило не было явно сформулировано, но обычно ожидается единственный
	спавн.)
- Пути к файлам текстур должны быть действительными (файлы доступны для
	чтения). (Это логичное требование, хотя в тексте прямо не указано.)
- Полосы пробелов внутри строк карты должны интерпретироваться как реальные
	символы карты — при необходимости строки карты приводятся к одной
	прямоугольной сетке (например, дополняются пробелами) перед проверкой
	замкнутости.
- Линии карты могут иметь разную длину в исходном файле; ваша парсер-логика
	должна корректно это обрабатывать.
- Допустимо наличие лишних пробелов/табуляций вокруг идентификаторов и
	значений элементов (но внутри пути к текстуре пробелы могут быть частью
	имени файла — следует аккуратно обрабатывать). Точное поведение зависит от
	реализатора парсера; лучше явно задокументировать в коде.
- Пробельные символы в конце/начале строк (trailing/leading spaces) не должны
	ломать парсер — их можно корректно игнорировать там, где это уместно
	(например, при чтении идентификаторов), но не внутри самой карты.
- Наличие пустых строк до/после элементов разрешено (для удобства
	форматирования), кроме того, карта может содержать пробелы как часть
	содержимого.
- При проверке замкнутости карты следует учитывать, что пробелы вне
	записанных строк карты (например, между блоками элементов) не являются
	её частью.

Если какие-то из подразумеваемых пунктов вы хотите сделать обязательными в
правилах — сообщите, и я внесу их в основной раздел как явные требования.
